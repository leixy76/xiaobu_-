TMR的快速会议 - 核心摘要

会议日期: 2025-06-18

核心参会人: Xuxun, Yinqi, Weitao, Yangdi, Jun, Oldhu

会议主题: 讨论和规划产品中AI能力（代号TMR或Copilot）的具体形态、交互方式、核心场景和实施优先级。

会议核心摘要 (TL;DR)

  • 最终共识： 确定采用一个全局、上下文感知的 “对话式Copilot” 界面作为AI交互的主要入口，而不是在各处分散地添加AI功能。这个界面需要有历史记录和多会话管理能力。
  • 产品形态： Copilot将整合现有产品中分散的入口（如Home、Inbox、通知），成为用户每日工作的起点，提供信息汇总和快速操作。它应设计为与主界面协同工作的侧边栏或可扩展面板，而非完全取代或打断现有流程。
  • 核心价值与场景优先级：
    1. 短期 (提升存量用户体验): 优先解决高频、琐碎的“脏活累活”，如：自动填写工时、一键生成周报/项目报告、信息摘要。
    2. 中期 (增强核心竞争力): 实现更复杂的软件工程任务，如：从需求文档(Wiki)一键拆解并创建工作项、生成测试用例。
    3. 长期 (吸引新客户/管理者): 提供面向管理者的风险监控、数据分析看板、安全与合规审计等高级功能。
  • 技术实现策略：
    • [Oldhu] 提出的核心观点被广泛认可：必须优先构建一套健壮的底层能力层(MCP - 多模态Copilot协议)，即一系列AI可调用的函数/API。这是实现一切上层交互的基础。
    • AI的引入将倒逼产品基础能力的完善，如：需要更强的基线(Baseline)对比、操作回滚、审批流等机制来承载AI操作的不确定性。
  • 下步行动： [Xuxun] 将基于本次讨论，从关键角色（PM、研发、管理者）和高频场景出发，细化Copilot的产品设计，包括快捷方式、组件能力和交互流程，并优先推进一些确定性强的场景落地。

一、 AI交互模型的核心设计与争论

1.1 从分散功能点到统一对话入口

  • [Xuxun] 初始想法是在不同页面（上下文）提供推荐的快捷指令，但发现自动判断用户意图的不确定性太高。
  • [Yinqi] 提出应该有一个**全局可交互的对话框**，在任何界面都能唤出。这是会议早期形成的核心共识。
  • [Oldhu] (关键战略观点) 指出，提供一个开放的对话框，本质上是创造了一种全新的、用户行为不可控的交互模式。这不仅仅是加功能，而是可能需要重塑整个产品的交互逻辑。用户会期望在这个对话框里完成所有事，这是一个巨大的投入。
  • 结论： 尽管投入巨大，但团队普遍认为统一的对话式Copilot是未来的方向，是“最优解”。

1.2 上下文感知与交互细节

  • [Yinqi] 强调上下文的重要性，类比电商客服会自动带入订单号。我们的Copilot在特定页面（如一个工作项）打开时，应自动将该对象的ID或链接作为上下文，并让用户可见、可编辑。
  • [Weitao, Yangdi] 讨论了具体的UI形态。倾向于类似VS Code Copilot的**侧边栏模式**，它与主界面并存，用户可以同时操作两边，形成协同工作的体验，而不是打断式的弹窗。
  • [Xuxun] AI的输出形式应该是多模态的，可以是文本、列表、链接，甚至是图表，需要在对话框内有合适的组件来承载这些结果。

1.3 对话历史与记忆

  • [Yinqi] 强烈主张对话需要有**持久化的历史记录和记忆**。用户不应每次都从零开始。
  • [Weitao] 建议可以像ChatGPT一样，支持创建多个独立的对话（Session），比如一个用于“Bug管理”，一个用于“版本规划”，保持不同任务的上下文独立。

二、 核心用户场景与价值定位

2.1 解决“不想干的脏活累活”（提升效率）

  • [Yinqi] 明确指出AI的核心价值之一是帮助用户完成琐碎、重复但必要的工作。
  • 场景1：填写工时 (高频痛点)。 [Yinqi] 提出，用户可以通过自然语言指令，如“把我今天完成的任务，工时平均分配一下”，系统自动完成填报。
  • 场景2：生成报告 (高价值)。 [Yangdi, Yinqi] 用户可以要求“帮我生成上周的周报”，AI会自动汇总其完成的工作项、代码提交等信息，生成报告初稿。这被认为是面向个人和管理者都非常有吸引力的功能。

2.2 赋能软件工程核心流程（提升质量和速度）

  • 场景3：需求拆解。 [Yinqi, Weitao] 这是被反复提及的“获客”级功能。可以基于一篇Wiki/PRD文档，AI自动识别层级关系，批量创建对应的史诗、特性和用户故事。
  • 场景4：生成测试用例。 [Xuxun] 基于一个需求描述，AI可以自动生成结构化的测试用例，并存入测试用例库。
  • 场景5：代码规范检查。 [Jun] 提出，可以建立一个公司内部的开发规范知识库，AI在辅助编程时，能遵循这些规范。

2.3 服务管理者（提供洞察与管控）

  • [Yinqi] 管理者（PM/PMO）可以通过AI快速获得洞察，例如“告诉我当前项目的风险点”、“哪些任务延期了”。
  • [Jun] (关键补充) 提出了**AI使用的安全与审计**视角。管理者需要一个看板，看到AI的资源消耗、谁在用、用量多少，以消除安全焦虑。这个功能对决策者非常有吸引力。

三、 技术实现与战略思考

3.1 MCP (Copilot协议) 作为地基

  • [Oldhu] 强调，在构建复杂的对话界面之前，核心是先构建好底层的**能力(Capabilities)**，并将其封装为AI可调用的函数集（即MCP）。这样，即使没有自己的UI，也能通过集成第三方工具（如Cursor）来验证能力。
  • 观点： 底层能力的健壮性，远比上层UI的华丽更重要。

3.2 AI 与现有系统能力的结合与倒逼

  • [Yangdi] 认为AI操作必须是可追溯、可回滚的。这意味着产品需要有更强的**基线对比(Baseline)**和**操作日志**功能。
  • [Yinqi] AI的引入会**倒逼**产品基础能力的升级。例如，为了支持AI的复杂操作，必须完善审批流、版本控制等企业级功能。
  • 观点： AI不是一个孤立的功能，它的落地需要整个产品体系的支撑和进化。

3.3 平台化 vs. 功能化

  • [Xuxun, Weitao] 讨论了是提供一个开放的AI平台让客户自己DIY，还是提供开箱即用的具体功能。
  • 结论： 针对国内客户，更倾向于提供**具体、明确、有价值的场景化功能**，而不是一个需要高技术能力才能使用的开放平台。客户更关心“这个场景你们怎么解决”，而不是“你们有什么可能性”。

四、 后续计划与优先级

  • 短期： 优先实现一些确定性高、价值明显的功能。
    • [Xuxun, Yinqi] 快速迭代出Copilot的雏形，先集成Wiki内的文本生成与总结能力。
    • [Xuxun] 探索“AI Home”的概念，作为用户登录后的默认首页，整合待办、通知和摘要信息。
  • 中期： 逐步扩展到更复杂的软件工程场景，如需求拆解和测试用例生成。
  • 并行： 持续建设和完善底层的MCP能力库，为所有上层应用提供支持。
